The Problem

AI tools fail silently. This one fixes itself

A wealth manager asking an AI to analyze a 10-K filing needs specific data: dollar amounts, contract dates, risk ceilings, competitive metrics. A generic prompt produces plausible but non-actionable answers. Most teams fix this by manually writing better prompts, a slow, human-in-the-loop process.

FinLoop removes the human from that loop. It runs a judge that scores the agent's answers, identifies what failed, and rewrites the extraction prompt, iteratively, automatically, with full observability into every decision made.

Architecture

Three agents, one loop

Extractor Answers wealth-manager questions using the current prompt
→
Judge Scores each answer: specificity, completeness, accuracy, utility (0–100)
→
Optimizer Analyzes failure patterns, rewrites the extraction prompt
↰

Loop repeats until average score ≥ 88/100 or max iterations reached

The Idea

Agents that know how to improve themselves

Most AI deployments treat the prompt as a fixed artifact, something a human engineer writes once and maintains manually. The core insight behind FinLoop is different: the evaluation signal that tells you an answer is bad is itself a learnable resource. If an agent can score its own output, it can also use that score to rewrite the instructions that produced the output.

This creates a self-closing loop. The agent isn't just executing a task; it's continuously improving its own strategy for executing the task, using its own judgment as the teacher. No labelled training data. No human prompt engineering between iterations. Just a measurable goal and a loop that runs until it's met.

The analogy is a senior engineer doing a code review and then fixing the architecture that produced the bad code, not just patching the symptoms. The difference is the agent does this autonomously, in seconds, at the cost of a few API calls.

The Loop Pattern
1
Execute: Run the agent on the task with the current strategy
2
Evaluate: A judge agent scores the output against a rubric
3
Optimize: An optimizer agent rewrites the strategy based on failures
4
Repeat: Until quality target is met, cost ceiling hit, or improvement plateaus
The same pattern generalizes to any task where you can define what "good" looks like. The judge is the key: if you can score it, you can loop it.
Applications in Wealth Management

Where self-improving loops change the economics

Vestmark manages infrastructure for $2T+ in assets and 5M+ accounts. At that scale, the difference between a prompt that extracts good insights and one that misses material data compounds across thousands of advisors and millions of decisions. Self-improving loops are most valuable wherever the task is repeated, the quality signal is measurable, and manual prompt maintenance doesn't scale.

10-K / Earnings Analysis

Optimize the prompt that extracts risk factors, guidance, and competitive signals from filings, at scale across every holding in a portfolio, not just the largest names.

Client Report Generation

Financial advisors send hundreds of portfolio review letters. A loop improves the generation prompt until reports are specific, compliant, and advisor-approved, without per-report human editing.

Rebalancing Rationale

Regulators require documented rationale for every trade. A loop optimizes the explanation agent until generated rationales consistently pass compliance review, reducing reviewer load at scale.

Risk Flag Classification

Classify portfolio anomalies as risk events or noise. A loop optimizes the classification prompt against historical labelled outcomes, improving precision without retraining a model.

RFP Response Quality

Enterprise RFPs require precise, defensible answers. A loop scores draft responses against evaluation criteria and refines the generation strategy until responses consistently win reviewer approval.

Regulatory Change Monitoring

Monitor SEC/FINRA rulemaking documents for client impact. A loop improves the impact-extraction prompt as new rule types emerge, without manual re-engineering after each regulatory change.

Common thread: each of these tasks is high-stakes, repeatable, and has a measurable quality signal. That's the precondition for a self-improving loop to replace manual prompt maintenance.

The Proposal

Loops should be infrastructure, not experiments

Most AI deployed in production today is static. An engineer writes a prompt, it works well enough, and it stays that way until someone notices it's degrading or the task changes. The loop is treated as a one-time setup problem, not an ongoing system.

My proposal is a different framing: the evaluation signal is the most valuable artifact you produce, more valuable than the answer itself. If you can score an output, and in most real-world tasks, you can define what good looks like, then you can loop. And if you can loop, the system improves itself automatically, without a human rewriting prompts after every edge case.

This shifts the engineer's job. Instead of writing better prompts, you design better evaluation rubrics. Instead of debugging bad outputs, you tune the feedback signal. The loop does the rest.

What changes when loops are infrastructure
Static deployment: prompt written once, maintained manually, degrades silently
Loop as infrastructure: system continuously evaluates and improves its own strategy
Human in the loop: engineer reviews bad outputs, edits prompt, redeploys
Rubric in the loop: judge agent flags failures, optimizer rewrites root cause
Scales with headcount: more edge cases = more engineer time
Scales with compute: more edge cases = more loop iterations

The constraint is the evaluation rubric. Writing a rubric that accurately captures what "good" means in a given domain is genuinely hard: it requires domain knowledge, clear thinking about edge cases, and iteration of its own. In financial analysis, "good" means specific, complete, accurate, and actionable for investment decisions. That four-dimension rubric was the real design work in this project; the loop mechanics are straightforward once the rubric is sound.

What I'd explore next: applying the same loop architecture to tasks where the strategy being optimized is not a prompt but a workflow, specifically a sequence of agent steps, tool calls, or routing decisions. The judge evaluates the outcome of the full workflow; the optimizer restructures the steps. That's a harder problem, but the same principle holds: if you can score it, you can loop it.
